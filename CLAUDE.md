# BTTR — Build, Test, Trash, Repeat

## What This Is

BTTR is a product management framework for the AI era. It replaces traditional prioritization frameworks like RICE by eliminating "Effort" and "Complexity" as meaningful variables — AI removes those constraints. What remains is **delivering the right thing**, and the fastest path to that is rapid iteration: build it, test it, trash it if it doesn't work, repeat.

## Core Thesis

Traditional PM frameworks (RICE, WSJF, ICE) all weight effort and complexity. When AI can prototype anything in minutes and has the technical depth to implement any concept, those variables collapse to near-zero. The only thing that matters is whether the thing you built is the *right* thing. You find that out by shipping, not by scoring.

**The BTTR cycle:**
1. **Build** — Ship a working version fast. No spec paralysis. AI builds, PM directs.
2. **Test** — Put it in front of real signals. Does it solve the problem? Does anyone care?
3. **Trash** — If it doesn't work, kill it without attachment. Sunk cost is near-zero.
4. **Repeat** — Take what you learned, build the next iteration or pivot entirely.

## Principles for This Project

- **Ship over spec.** If we can build it faster than we can describe it, build it.
- **Trash without guilt.** Killing something that doesn't work is progress, not waste.
- **Bias toward concrete.** Prototypes > decks. Working code > wireframes.
- **Stay opinionated.** BTTR has a point of view. We're not writing a balanced survey of PM frameworks — we're arguing that the old ones are broken.

## Content & Messaging Notes

- BTTR is a framework, not a practice or methodology.
- **"The Better Framework"** — BTTR is read/spoken as "better." The tagline and the acronym are the same word. Use this everywhere: "the BTTR framework" = "the better framework." It's the name and the claim in one.
